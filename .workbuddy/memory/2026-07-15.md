## 排行榜乱码问题诊断与修复

**现象**: 前端排行榜显示番茄平台书名/作者乱码，如"0挂0打0口"等PUA区字符

**根因确认**:
- 番茄小说排行榜页面使用字体反爬机制，将中文字映射到PUA区(U+E000~U+F8FF)
- fanqie.js 已有两步抓取策略（榜单→详情页），但合并逻辑 `detail.bookName || rank.bookNameRaw` 在详情页失败时 fallback 到了乱码raw数据
- 数据库 mochao_total.sql 中 hot_search 榜(ID 25-34) 全部是乱码数据，new_book 榜(ID 35-44) 是正常明文

**修复措施**:
1. **scraper/src/fanqie.js**: 新增 `hasPuaChars()` PUA检测函数，在提取raw字段时过滤乱码、在合并时再次过滤，绝不让乱码数据入库
2. **backend/RankingServiceImpl.java**: 新增 `hasPuaChars()` Java版检测，保存快照时过滤含PUA字符的条目
3. **backend/RankingServiceImpl.java**: 新增 `cleanGarbledData()` 清洗方法，扫描全表→删除乱码记录→清除Redis缓存
4. **backend/RankingController.java**: 新增 `POST /v1/ranking/clean-garbled` 管理员清洗API
5. 七猫(qimao.js)/刺猬猫(ciweimao.js) 确认无字体反爬问题（SSR HTML直解析）

**受影响数据**: 番茄平台 hot_search/read_rank 榜单 2026-07-13 数据（约20条乱码记录）

## 乱码自动自愈机制

**需求**: 系统自行处理乱码记录，无需管理员手动调用API

**实现三层自愈机制**:
1. **查询层自愈** — `querySnapshots()` 发现乱码时立即过滤 + 异步触发清洗重抓
   - 用户查询时立即看到干净数据（乱码条目被过滤）
   - `CompletableFuture.runAsync()` 后台执行 `cleanGarbledDataFor()` + `forceReScrape()`
   - 不阻塞用户请求

2. **抓取层自愈** — `triggerScrape()` 检测到已有数据含乱码时强制覆盖重抓
   - 原逻辑: `checkTodayData()` 返回 true → 跳过
   - 新逻辑: 返回 true 但 `hasGarbledData()` 返回 true → 清除旧数据 + 重新抓取
   - 返回 true 且无乱码 → 正常跳过

3. **定时巡检** — `RankingScheduler` 每天 4:00 自动扫描所有榜单
   - 在 3:00 抓取完成后 1 小时执行
   - 对每个 platform+rankType 调用 `hasGarbledData()` 检测
   - 发现乱码 → `selfHealAsync()` 异步清洗+重抓

**关键改动文件**:
- `MochaoApplication.java`: 新增 `@EnableAsync`
- `RankingService.java`: 新增 `hasGarbledData()`/`forceReScrape()`/`selfHealAsync()` 接口
- `RankingServiceImpl.java`: 实现三层自愈逻辑 + `cleanGarbledDataFor()` + `invalidateRedisCache()`
- `RankingScheduler.java`: 新增 `autoCleanGarbled()` 定时巡检（cron 4:00）
- `RankingController.java`: `/clean-garbled` API 注释更新为"管理员兜底操作"
