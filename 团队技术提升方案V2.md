## 四、代码规范与质量标准

### 4.1 Java 后端规范

#### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase | `NovelServiceImpl`, `AiFeatureType` |
| 方法名 | camelCase，动词开头 | `findByUserId`, `callAiWithRetry` |
| 常量 | UPPER_SNAKE_CASE | `MIN_SECRET_LENGTH`, `MAX_RETRY_COUNT` |
| 枚举值 | UPPER_SNAKE_CASE | `CHAPTER_GENERATE`, `OUTLINE_EXPAND` |
| DTO字段 | camelCase | `userId`, `chapterContent` |
| 包名 | 全小写，按模块 | `module.ai.service.impl` |

#### 方法设计规范

```
✅ 好的方法：
- 单一职责，一个方法做一件事
- 参数 ≤ 4 个（超过用 DTO/对象封装）
- 返回值明确，不用 Map<String, Object>（除非必要）
- 异常用 BusinessException + ResultCode，不用 raw RuntimeException

❌ 避免：
- 方法超过 50 行（拆分）
- 参数超过 4 个（封装 DTO）
- 在 Controller 里写业务逻辑
- 用 Map 代替专门的结果对象
```

#### 安全编码清单

```java
// ✅ 所有密码/密钥必须通过环境变量注入，不允许明文兜底
@Value("${jwt.secret}")  // 没有 :default-value
private String jwtSecret;

// ✅ 所有 API 端点必须有明确的权限标注
@PreAuthorize("hasRole('ADMIN')")  // 或 @Secured
public Result adminOp() { ... }

// ✅ 所有用户输入必须校验
public Result create(@Valid @RequestBody NovelCreateDTO dto) { ... }

// ✅ 数据库操作防并发
@Update("UPDATE novel SET chapter_count = chapter_count + 1 WHERE id = #{id}")
void incrementChapterCount(@Param("id") Long id);  // 原子操作，不是 read-then-write
```

#### 测试规范

```java
// ✅ 测试命名：方法名_场景_预期结果
@Test
void callAiWithRetry_业务异常不重试_直接抛出() { ... }

@Test
void parseClaims_有效Token_返回正确Claims() { ... }

// ✅ 测试结构：Given-When-Then
@Test
void saveLog_枚举Feature_使用枚举Code() {
    // Given
    AiFeatureType feature = AiFeatureType.CHAPTER_GENERATE;
    
    // When
    AiUsageLog log = service.saveLog(userId, novelId, chapterId, feature, ...);
    
    // Then
    assertThat(log.getFeature()).isEqualTo("chapter_generate");
}
```

### 4.2 Vue 前端规范

#### 组件设计规范

```
✅ 好的组件：
- 单文件组件 < 150 行（超过拆分为子组件）
- Props 有类型声明和默认值
- 事件命名 kebab-case，语义清晰
- 计算属性替代模板中的复杂逻辑
- Vuex action 处理异步，mutation 处理同步

❌ 避免：
- App.vue 承载过多逻辑（当前 8191字节需拆分）
- 组件内直接调 API（应通过 Vuex action）
- v-if 和 v-for 同元素使用
- 深层嵌套的 watch（用 computed 替代）
```

## 九、从"资深开发者"视角的建议

作为高级开发工程师，我在评审中看到的不仅是问题，更是**团队的成长潜力**：

### 9.1 值得肯定

- **架构直觉不错** — 模块化分层、构造器注入、原子更新这些说明团队有良好的设计意识
- **安全意识正在觉醒** — JWT加固、限流、环境变量化都在往正确方向走
- **愿意接受重构** — Scraper 从 2149 行拆到模块化，说明团队不抵触改动

### 9.2 最需要改变的习惯

1. **"先写功能，测试以后补"** → 改为 **"先写测试，再写功能"**  
   TDD 不是负担，是安全网。测试先行让重构有底气。

2. **"配置写死方便调试"** → 改为 **"环境变量 + .env 模板"**  
   明文凭据是定时炸弹。已经修复了 dev.yml，但团队每个人都要养成习惯。

3. **"大文件拆分太麻烦"** → 改为 **"超过200行就该拆"**  
   维护成本是随文件长度指数增长的。Scraper 已经拆了，后端/前端也要保持这个纪律。

4. **"Review 是走流程"** → 改为 **"Review 是学习机会"**  
   好的 Review 不是挑毛病，是互相学习。通过 Review 传播最佳实践比文档有效 10 倍。

### 9.3 一句话总结

> **测试筑底 → 性能提升 → 架构深化 → 技术栈升级**  
> 四步走，每一步都有测试兜底，每一步都有量化指标验证。不追求一步到位，追求**每周都有可衡量的进步**。
